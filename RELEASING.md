# Releasing agentberg

Pushing a version tag (`v*`) triggers `.github/workflows/publish.yml`:

- **GitHub Release — always.** Created from the matching `kit_manifest.json` entry.
- **PyPI — opt-in.** The publish job only runs when the repo variable
  `PUBLISH_TO_PYPI` is set to `true` (Settings → Secrets and variables → Actions →
  Variables). Until then, releases are **GitHub-only** — no PyPI version is burned.

## Enabling PyPI later (one-time)

PyPI publishing uses **Trusted Publishing (OIDC)** — no API token or secret to manage.
When you're ready to publish to PyPI:

1. Sign in at [pypi.org](https://pypi.org) → **Your projects** → **Publishing** →
   **Add a pending publisher**.
2. Fill in exactly:
   - **PyPI Project Name:** `agentberg`
   - **Owner:** `ganeshnallasivam-cell`
   - **Repository name:** `agentberg-starter`
   - **Workflow name:** `publish.yml`
   - **Environment name:** `pypi`
3. Save. (The project is created on first successful publish.)
4. Set repo variable `PUBLISH_TO_PYPI=true` (Settings → Secrets and variables →
   Actions → Variables). The next tag push will then publish to PyPI as well.

That's it — no token is ever stored in GitHub.

## Release notes — the rule (non-negotiable)

**Every release ships release notes. No exceptions.** The single source of truth is the
`changelog` array in `kit_manifest.json`. Everything else is derived from it, so nothing
can drift:

- `CHANGELOG.md` — human-readable, generated by `scripts/release_notes.py --write`.
- **GitHub Release** — created automatically on every tag push from the matching
  manifest entry (`github_release` job in `publish.yml`).

Three guards enforce this so a release can't go out without notes:

1. **CI** (`ci.yml`) fails if `CHANGELOG.md` is out of sync with the manifest.
2. **Publish** (`publish.yml`) fails the build if there is no `changelog` entry for the
   tag's version, and re-checks `CHANGELOG.md` sync.
3. The GitHub Release is published automatically — you never hand-write it.

## Cutting a release

The git tag must match the version in `pyproject.toml` (the workflow enforces this), and
`kit_manifest.json` must contain a `changelog` entry for that version.

```bash
# 1. add the changelog entry FIRST
#    edit kit_manifest.json: bump top-level "version" + prepend a new changelog entry
#    {version, date, files, added: [...]}

# 2. regenerate the human changelog from the manifest
python scripts/release_notes.py --write

# 3. bump the package version to match
#    edit pyproject.toml:           version = "2.0.0"
#    edit agentberg_cli/__init__.py: __version__ = "2.0.0"

# 4. commit everything together
git commit -am "Release v2.0.0"

# 5. tag and push
git tag v2.0.0
git push origin main --tags
```

Pushing the tag triggers the workflow: it builds the sdist + wheel, checks the tag matches
`pyproject.toml`, verifies release notes exist and are in sync, smoke-tests
`agentberg --help`, **creates the GitHub Release from the manifest entry**, and — only if
`PUBLISH_TO_PYPI=true` — publishes to PyPI.

Within a minute or two:

```bash
pipx install agentberg        # or: uv tool install agentberg
```

## Notes

- **Single source of truth.** The kit and the CLI live in this one repo; the package
  is *built from here*, so there is no separate copy to drift.
- **`kit_manifest.json` and the package version are kept in lockstep.** The manifest
  drives pull-to-review for the *kit files* an installed agent adopts; the PyPI version
  is the *CLI tool* itself. They were historically independent, but as of v2.0.0 a
  release bumps both to the same number so the tag, the GitHub Release, the changelog
  entry, and the published CLI all line up.
- **CI** (`ci.yml`) builds and smoke-tests on every push/PR, so a broken build is
  caught before you tag.
